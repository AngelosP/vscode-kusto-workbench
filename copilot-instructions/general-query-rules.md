# General rules about writing Kusto queries

* When using dcount(), always use **dcount(, 2)** precision.
* When working with date ranges, favor **Date >= startDate** and **Date < endDate** so that the parameters startDate and endDate can both be start of days.
* It is very likely that the Kusto database contains a Kusto function or a Kusto table that is suitable for the task at hand, so **always check for one first** before trying to write out the Kusto logic from scratch each time. If you have a choice between a function and a cooked Kusto table, prefer to Kusto cooked table over the function unless explicitely told otherwise.
* When you are asked to calculate something, like for example 'calculate the number of active users', or 'calculate the retention', it doesn't mean you cannot use pre-cooked tables that contain the number. Users typically don't know what is already cooked and what isn't, they just want the number but they are not very accurate with their english so they say 'calculate' when they really mean 'get the number'. So always favor pre-cooked tables and views over doing the calculation from scratch. Prefer them over Kusto functions as well. Power users who want you to come up with the calculation on your own from scratch will explicitely tell you to avoid using pre-cooked tables in general or they will explicitely tell you which tables to avoid.
* You have access to a tool called '**Execute Kusto query and read the results'**\. Use it to check that your response to the user request is the best possible it can be\, and it doesn't suffer from assumptions or incosistencies that turn out to be incorrect once the query executes\. For example\, use it to retrieve the implementation of Kusto functions before using them to understand how they work\, and use it to retrieve data from tables and views so that you can understand what data they contain before writing or fixing queries that use them \(e\.g\. \| take 10 to get the first few rows\, or \| distinct \<columnName\> to get the unique values of a column\, etc\.\)\.
* Some databases contain only raw telemetry data and they lack cooked tables and / or useful Kusto functions. It's not a certainty, but often these databases contain tables with 'Raw' in their name, or other generic names like 'IncomingData', and they might have complex columns that are JSON or other complex datatypes. Explore these databases and tables carefully using the '**Execute Kusto query and read the results'** tool because they will often contain a treasure trove of data, they are just not going to be in explicit columns and hide behind complex datatype columns with names like 'JSON', 'Properties', 'Measures', 'Values', 'Results', etc.
* Always try to come up with excellent performing queries, even if the task is not explicitely to optimize an existing query. Especially if your response is a Kusto query that is complex, or spans a wide range of time, or does joins with many rows make sure you are doing things effeciently. You have access to a tool called '**Get query optimization best practices**', use it to make sure you are following best practices when necessary even if the task is not explicitely to optimize an existing query.

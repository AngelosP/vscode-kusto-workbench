# General rules about writing Kusto queries

* When using dcount(), always use **dcount(, 2)** precision.
* When working with date ranges, favor **Date >= startDate** and **Date < endDate** so that the parameters startDate and endDate can both be start of days.
* It is very likely that the Kusto database contains a Kusto function or a Kusto table that is suitable for the task at hand, so **always check for one first** before trying to write out the Kusto logic from scratch each time. If you have a choice between a function and a cooked Kusto table, prefer the Kusto cooked table over the function unless explicitely told otherwise.
* When you are asked to calculate something, like for example 'calculate the number of active users', or 'calculate the retention', it doesn't mean you cannot use pre-cooked tables that contain the number. Users typically don't know what is already cooked and what isn't, they just want the number but they are not very accurate with their english so they say 'calculate' when they really mean 'get the number'. So always favor pre-cooked tables and views over doing the calculation from scratch. Prefer them over Kusto functions as well. Power users who want you to come up with the calculation on your own from scratch will explicitely tell you to avoid using pre-cooked tables in general or they will explicitely tell you which tables to avoid.
* You have access to a tool called '**Execute Kusto query and read the results**'. Use it liberally to check that your response to the user request is the best possible it can be, and it doesn't suffer from assumptions or inconsistencies that turn out to be incorrect once the query executes. For example, use it to retrieve the implementation of Kusto functions before using them to understand how they work, use it to retrieve data from tables and views so that you can understand what data they contain before writing or fixing queries that use them (e.g. `| take 10` to get the first few rows, or `| distinct <columnName>` to get the unique values of a column, etc.), and use it to check the date range of tables and functions before committing to use them.
* **Before using any function or table**, verify it covers the user's requested date range by querying its actual min/max dates (e.g., `| summarize min(DateColumn), max(DateColumn)`). Do NOT assume a data source has the data you need, confirm it first. If it doesn't cover the full range, do not use it and apologize later; instead, search for alternatives before presenting results.
* **When a data source doesn't have enough data** (wrong date range, missing columns, etc.), do NOT give up and tell the user "that's all we have". Instead, actively search for alternatives: use `.show tables` to find related tables, check for raw/source/archive tables that may have longer retention, examine the function's implementation with `.show function FunctionName` to find its underlying tables, and check the date ranges of all candidate tables. Only after exhaustively searching the database should you tell the user the data isn't available.
* Some databases contain only raw telemetry data and they lack cooked tables and / or useful Kusto functions. It's not a certainty, but often these databases contain tables with 'Raw' in their name, or other generic names like 'IncomingData', and they might have complex columns that are JSON or other complex datatypes. Explore these databases and tables carefully using the '**Execute Kusto query and read the results**' tool because they will often contain a treasure trove of data, they are just not going to be in explicit columns and hide behind complex datatype columns with names like 'JSON', 'Properties', 'Measures', 'Values', 'Results', etc.
* Always try to come up with excellent performing queries, even if the task is not explicitely to optimize an existing query. Especially if your response is a Kusto query that is complex, or spans a wide range of time, or does joins with many rows make sure you are doing things effeciently. You have access to a tool called '**Get query optimization best practices**', use it to make sure you are following best practices when necessary even if the task is not explicitely to optimize an existing query.
* Leave yourself 'development notes' using the tool '**Update development note**' that help you remember things you discovered about the data that will save you time to read in the future before doing new work. You can use it to add a new note, update an existing one, or remove one (just pass blank content to remove it). When a new conversation starts, all the development notes from the current file will be shared with you at the start of the conversation.
